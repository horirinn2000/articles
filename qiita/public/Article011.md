# はじめに

ReactやVue.jsなどのSPA（Single Page Application）でアプリケーションを更新・デプロイしたのに、ユーザーから「画面が古いままです！」と言われた経験はありませんか？
何度リロードしても変わらず、スーパーリロード（Ctrl+Shift+R）をして初めて新しい画面が表示される...。

この問題の多くは、フロントエンドのビルドの仕組みと「ブラウザのキャッシュ」が噛み合っていないために起こります。

# なぜソース改修がすぐに反映されないのか？

SPAのビルドツール（Vite, Webpackなど）は、ビルドのたびにJSやCSSのファイル名にハッシュ（ランダム文字列）を付与します。
* 例： `main.a1b2c3d4.js` -> 改修後 -> `main.x9y8z7w6.js`

ハッシュ値が異なるため、意識しなくても、ビルドの度に再度読み込まれるはずです。
しかし、新しいJSファイルがサーバーに配置されても、ブラウザが最初に読み込むエントリーポイントである **`index.html`** がキャッシュされているとどうなるでしょうか？
ブラウザはキャッシュされた古い `index.html` を使い、その中に書かれている **古いハッシュ値のJS** (`main.a1b2c3d4.js`) を読み込もうとします。

結果として、新しいJSの読み込みが走らず、画面が古いままになってしまうのです。

# 罠となるRFCの「10%ルール」（ヒューリスティックキャッシュ）

「いや、Nginx側でキャッシュの有効期限（Expiresなど）を特に設定していないから、毎回最新を取りに行くはずだ」
そう思っていると罠にハマります。

Webの標準規格（RFC 7234）には、サーバー側から明示的な `Cache-Control` ヘッダが送られてこなかった場合のブラウザの標準的な振る舞いとして **ヒューリスティックキャッシュ（Heuristic Caching）** が定義されています。
通称「10%ルール」とも呼ばれ、**「ファイルの最終更新日時（Last-Modified）から現在までの経過時間の約10%」に相当する期間、ブラウザが勝手にキャッシュを有効にしてしまう**という仕様です。

そのため、「サーバー側でキャッシュコントロールを設定していない」=「キャッシュされない」**ではありません**。結果としてブラウザの勝手な判断により `index.html` がキャッシュされ、デプロイ直後に新しいJSが読み込まれない事故が発生するのです。

# 解決策：Nginxで正しいCache-Controlを設定する

この問題を解決するためのベストプラクティスは、Nginx側で**HTMLは絶対にキャッシュさせず、JS/CSSなどの静的アセットは長期間キャッシュさせる**という設定を明示的に行うことです。

以下がその設定例です。

```nginx
server {
    listen 8080;
    root /api/public;
    index index.html;
    charset utf-8;

    # HTML のキャッシュを無効化（JS の変更を即座に反映）
    location / {
        root /api/public;
        try_files $uri $uri/ /index.html;
        # 毎回必ずサーバーに最新か確認させる。勝手なキャッシュを防ぐ
        add_header Cache-Control "no-store, no-cache, must-revalidate";
    }

    # JS / CSS / 画像のキャッシュを有効化（リネームで新しいファイルを取得）
    location ~* \.(?:js|css|png|jpg|jpeg|gif|ico|svg|woff2?|ttf|otf|eot)$ {
        # 1週間キャッシュし、再検証もさせない (immutable)
        add_header Cache-Control "public, max-age=604800, immutable";
    }

    # 圧縮された静的ファイルにもキャッシュ適用（gzip / brotli 対応）
    location ~* \.(?:js|css)\.(?:gz|br)$ {
        add_header Cache-Control "public, max-age=604800, immutable";
    }
}
```

## 設定の解説

1. **HTMLのキャッシュ無効化 (`location /`)**
   `Cache-Control "no-store, no-cache, must-revalidate"` を付与することで、ブラウザに `index.html` を一切キャッシュさせず、アクセスするたびに必ずサーバーから最新のHTMLを取得させます。
   これにより、デプロイ直後でも「最新のJSのハッシュ値が記載されたHTML」が確実に読み込まれるようになります。

2. **静的アセットの強力なキャッシュ (`location ~* \.(?:js|css|...)$`)**
   JSやCSSなどは、ソースコードが変更されるたびにビルドツールによってファイル名（ハッシュ値）が変わります。
   つまり、同じファイル名で中身だけが更新されることはありません。そのため、長期間ブラウザにキャッシュさせてネットワーク通信を減らしても安全です。`immutable` をつけることで、再読み込み時などの不要な再検証リクエスト（304 Not Modified の確認など）すら送らないようになり、パフォーマンスが向上します。

# 補足：Next.js などのフルスタックフレームワークを利用する場合は？

React単体でのSPA（Viteなどでビルドした静的ファイルをNginx等で配信する構成）ではなく、**Next.js** などのフルスタックフレームワークを利用している場合、キャッシュ戦略は大きく異なります。

結論から言うと、**Next.jsでは開発者が意図的にNginx等でキャッシュ設定を作り込む必要は（基本的には）ありません。**

Next.js（Node.jsサーバーやVercelなど）は、レスポンスを返す際に**フレームワーク側で自動的に最適な `Cache-Control` ヘッダを付与**してくれます。

* **静的アセット (`/_next/static/` 配下のJS/CSSなど)**:
  自動的に `Cache-Control: public, max-age=31536000, immutable`（1年間キャッシュ）がセットされ、強力にキャッシュされます。ファイル名にはハッシュが含まれるため、変更時には必ず新しいファイルとして読み込まれます。
* **Client Component (`"use client"` を指定したコンポーネント)**:
  App Router 等で `"use client"` を指定した部分の JS ロジック自体も、ビルド時にチャンク（ハッシュ付きJSファイル）として分割され、上記の「静的アセット」と同様に配信されます。そのため、1年間の強力なキャッシュが適用されつつも、デプロイによって中身が変わればハッシュが変わり、即座に最新ロジックが安全に読み込まれます（ハイドレーション時のミスマッチなどの心配はありません）。
* **HTMLドキュメントやAPIレスポンス**:
  SSR（サーバーサイドレンダリング）のページや、常に最新を保つべき動的なページでは、キャッシュさせない設定（`private, no-cache, no-store, max-age=0, must-revalidate` など）が自動的に適用されます。
* **ISR（Incremental Static Regeneration）のページ**:
  CDNやNext.jsサーバーのキャッシュ機構を活用するため、`s-maxage` や `stale-while-revalidate` といった高度なキャッシュヘッダが自動出力され、バックグラウンドでの再生成とうまく連携します。

このように、Next.jsはページの種類（SSG, SSR, ISR）やファイルの種類に応じて、パフォーマンスが最も高く、かつ古い画面が表示されないように緻密なキャッシュヘッダを出し分けてくれます。
逆に、前段にNginxやCDNを配置している構成の場合、Nginx側で勝手に `Cache-Control` ヘッダを強制的に上書きしてしまうと、**Next.jsの緻密なキャッシュ戦略（ISRなど）が破壊されてしまう**ため注意が必要です。Next.jsを利用する場合は、フレームワークが出力するヘッダをそのまま透過させる（上書きしない）のがベストプラクティスです。

# まとめ

SPAのデプロイで「更新されない！」というトラブルを防ぐには、インフラ側での意図的なキャッシュコントロールが必須です。

* ブラウザの「10%ルール」により、未指定の場合は勝手にキャッシュされる
* それを防ぐため `index.html` は絶対にキャッシュさせず、常に最新を取得させる
* `js` や `css` などのハッシュ化されたアセットは、長期間キャッシュさせてパフォーマンスを上げる

この基本原則を理解して、スムーズなフロントエンドのデプロイ環境を構築しましょう。
