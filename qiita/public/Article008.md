# はじめに

前回の記事では、「Log Analytics」を使ってログをSQLで分析する方法を解説しました。
しかし、Cloud Loggingには運用上気をつけなければならない**一つの仕様**があります。

それは、**「デフォルトのログバケット（`_Default`）は、30日でログが消去される」** ということです。
「先月リリースした機能の不具合報告が今日来た。ログを見よう」
そう思って検索しても、**31日前のログは既に跡形もなく消えています。**

今回は、この課題を解決するための「コストと利便性を両立した最強のログ管理戦略」を解説します。

1. **Cloud Logging の保存期間をあえて少しだけ「延ばす」戦略**
2. **Cloud Storage (GCS) に安価に「退避」するログシンク**
3. **GCSの過去ログを BigQuery で直接分析する「敗者復活戦」**

この3段構えで、**「直近は快適に、過去ログは激安に管理する」** 鉄壁の体制を構築しましょう。

# 1. ログの保存期間は延ばすべきか？（Cloud Loggingの仕様）

標準バケット（`_Default`）の保持期間は**30日間**です。
これを知らずに運用していると、月を跨いだバグ調査や監査対応で詰む危険性があります。

## 保存期間は最大10年まで延長可能

この期間は設定により変更可能です。
Cloud Console の **[ログストレージ] > [バケットの設定]** から、最大で **3650日（10年）** まで延長できます。

### 戦略：あえて「60日〜90日」に設定する

30日を超えた分のログは、**$0.01 / GiB / 月** のストレージ料金が発生します。
少し費用はかかりますが、実際の開発現場では「1〜2ヶ月前のログを見たい」というニーズが意外と多く発生します。

* **30日 (デフォルト)**: コスト最安。ただし、先々月の月末に起きたエラーを翌月に調査する際に困る。
* **60日〜90日 (おすすめ)**: **開発体験（DX）への賢明な投資**。追加コストは微々たるものであり、過去ログを別システムから復元するエンジニアの「調査工数」と比較すれば圧倒的に安上がりです。
* **1年以上 (非推奨)**: 検索頻度が極端に低いのに、比較的高価なCloud Loggingのストレージに置き続けるのはコストの無駄遣いです。

「直近の利便性」のために少し延ばすのはアリですが、**「不安だから念のため全部そのまま残す」という思考停止はNG**です。

# 2. 長期保存の最適解：Cloud Storage (GCS) へ逃がす

「安く」「長く（1年以上）」保存するための正解は、**Cloud Storage (GCS)** へのエクスポートです。

## ログルーター（Log Router）を用いた自動退避

ログルーターの「シンク」機能を使えば、コードを一切書くことなく自動的にGCSへログを転送し続けることができます。

1. **保存先GCSバケットの作成**:
   * ストレージクラスは **Archive（アーカイブ）** を強く推奨します。
   * Archiveクラスの料金は **$0.0012 / GiB / 月** と圧倒的な安さを誇り、Cloud Loggingの超過ストレージ料金（$0.01）の**約1/10**のコストで済みます。
2. **ログルーターでシンクを作成**:
   * シンクの宛先: `Cloud Storage バケット`
   * バケットを選択: 1.で作ったArchiveバケットを指定
   * 包含フィルタ: 全て残すなら空欄、エラーのみ残すなら `severity >= ERROR` などを指定

これだけで、ログは自動的にGCSにJSONファイルとして出力され続けます。
Archiveクラスを利用すれば、仮に1TB（1000GB）保存し続けても月額1.2ドル（約180円）レベル。これでもう「昔のログが消えてしまった！」という恐怖から完全に解放されます。

# 3. GCSに置いたログの活用法（BigQuery外部テーブル）

「GCSにファイルとして置いたログなんて、テキスト検索しにくいしゴミ捨て場同然では？」
そう思われるかもしれませんが、実はGCS上のログは将来的な分析のための「宝の山」になります。

ここで活躍するのが、BigQueryの **外部テーブル（External Table）** という超強力な機能です。

## 外部テーブルによる「敗者復活戦」

BigQuery内にデータをわざわざインポート（コピー）しなくても、**GCSに転がっているJSONファイルを直接、BigQueryのSQLで検索・集計できる**機能です。

### 手順とクエリイメージ

1. **BigQuery でテーブルを作成**
   * テーブル作成元: `Google Cloud Storage`
   * ソースURI: `[cloud storage bucket]/run.googleapis.com/stdout/*`
   * ファイル形式: `JSONL（改行区切り）`
   * テーブル名: `log_external_table`
   * テーブルタイプ: `外部テーブル`

以上の設定だけで、すぐに以下のようなSQLが実行可能になります。
```sql
SELECT
  *
FROM
  `[my_project].[my_dataset].log_external_table(テーブル名)`
WHERE
  severity = 'ERROR'
  AND timestamp BETWEEN '2026-03-04 09:00:00' AND '2026-03-04 10:00:00'
ORDER BY timestamp DESC
LIMIT 10
```

わざわざ重いバッチ処理でログを復元しなくても、**「見たい時に、置いたままクエリを叩く」** ことができるのです。

# 4. 結論：コストと利便性を極めた「ログのライフサイクル戦略」

これまでの内容を統合すると、シニアエンジニアが採用すべき**「最強のログ戦略」**は以下のようになります。

### Step 1: 直近（0日〜90日）
**→ Cloud Logging (Log Analytics) で高速分析**
* 開発体験（DX）を重視し、保持期間を**あえて少し長め（60〜90日）に設定**します。
* ログはCloud Loggingに存在するため、Log AnalyticsのSQLを使って爆速で調査が可能。
* 多少のストレージ課金は「エンジニアの快適料」として割り切ります。

### Step 2: 長期保存（91日〜数年）
**→ GCS Archive へ自動退避**
* 検索頻度が落ちたログは、最安クラスの GCS Archive に流し込みます。
* コンプライアンス対応や監査対応のための「お守り」としての保管が主目的です。

### Step 3: イレギュラーな過去ログ調査
**→ BigQuery 外部テーブルで直接クエリ**
* 「半年前の特定のログを調査したい」といった緊急時（敗者復活戦）にのみ使用します。
* GCSに置いたままSQLで検索できるため、一時的な調査に最適です。

> **💡 Tips: いつBigQueryへのネイティブ保存を使うべきか？**
> 「過去1年分のログを使って頻繁に機械学習の学習データを抽出したい」「毎日ダッシュボードで過去ログを集計したい」といった**高頻度・大量スキャン**が必要なユースケースに限り、ログシンクの宛先を直接BigQueryにする（ネイティブ保存）設計を検討してください。外部テーブルのクエリスキャンは都度JSONを読み込むため、高頻度な検索にはコストと速度面で不向きです。

# まとめ

* **30日の壁**: デフォルトは30日だが、運用の都合に合わせて **60〜90日に延ばす** のが現場目線では賢い選択。
* **長期はGCSへ**: 1年以上の保管は `Archive` クラスのGCSバケットへ。圧倒的な低コスト（約1/10）を実現。
* **外部テーブルを活用**: GCSに置いたログはゴミではない。BigQueryから直接SQLで検索できる。

「直近は Log Analytics で快適に、過去ログは GCS + BigQuery で賢く管理する」。
この使い分けができれば、コスト最適化とエンジニアの利便性を高次元で両立した、強靭なログ基盤が完成します。
